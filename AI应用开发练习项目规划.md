# AI 应用开发练习项目规划

> 整理日期：2026-07-12  
> 用途：在新的 Codex 会话中继续讨论技术选型、架构设计和实施计划。

## 1. 个人背景与目标

- 软件工程专业应届毕业生，目前从事 Java 开发。
- 希望从传统 Java 开发逐渐转向 AI 应用开发，而不是模型训练或算法研究。
- 已在 Codex 辅助下，仅参考官方教程完成过 GLM-5.2 量化模型部署。
- 本地计算资源有限，计划通过大模型 API 完成应用，不依赖本地模型推理。
- 倾向以实践推动学习：先独立完成项目，再与成熟开源项目对比，理解产品设计、架构和工程实现上的差异。
- 可以接受陌生语言和框架，但技术选型应服务于学习目标，不为了使用新语言而增加无关复杂度。

## 2. 项目定位

计划从零开发一个：

> **基于大模型 API 的知识库与任务助手**

项目不以训练模型为目标，而是练习如何把大模型、业务数据和软件系统结合，形成一个完整、可使用、可评估的 AI 应用。

长期定位不是简单的聊天界面，而是一个能够：

- 接入不同大模型服务；
- 使用用户资料回答问题；
- 给出可以核查的内容来源；
- 保存会话和调用记录；
- 后续通过工具调用完成受控任务；
- 对质量、耗时、错误和费用进行观察的应用。

## 3. 第一版目标（MVP）

第一版只完成最小业务闭环，避免过早加入复杂的 Agent 和工作流。

### 3.1 核心流程

```text
配置模型服务
    ↓
创建知识库并上传资料
    ↓
解析、切分和索引文档
    ↓
用户提出问题
    ↓
检索相关资料（RAG）
    ↓
调用大模型 API
    ↓
流式返回带来源的回答
    ↓
保存会话、调用记录和用户反馈
```

### 3.2 第一版功能范围

- 接入至少一家兼容 OpenAI 风格接口或提供正式 SDK 的模型服务。
- 模型调用与具体供应商解耦，为以后切换模型保留接口。
- 创建、查看和删除知识库。
- 上传一种或少量常见格式的文档，例如 Markdown、TXT 或 PDF。
- 完成文档解析、分段、向量化和检索。
- 基于检索结果生成回答，并展示引用来源。
- 支持流式响应。
- 保存会话与消息历史。
- 记录模型名称、响应耗时、Token 用量、调用状态和错误信息。
- 允许用户对回答进行简单的有用/无用反馈。
- 提供最基本的错误提示和重试能力。

### 3.3 第一版暂不实现

- 多智能体协作。
- 可视化工作流编排器。
- 插件市场。
- 大量模型供应商适配。
- 复杂多模态能力。
- 自动浏览器操作或高风险外部动作。
- 企业级多租户、计费和组织管理。
- 为追求“完整平台”而复制 Dify 等项目的全部功能。

## 4. 后续可选迭代

完成 MVP 并复盘后，再根据兴趣逐步选择：

1. 增加第二家模型供应商，验证模型抽象是否合理。
2. 增加工具调用，例如查询数据库、生成报告或调用一个受控业务接口。
3. 建立固定测试问题集，比较不同提示词、切分策略、检索参数和模型的效果。
4. 增加调用追踪、质量评估和费用统计。
5. 增加文档重新处理、增量更新和失败任务重试。
6. 增加用户、权限和知识库隔离。
7. 再考虑简单工作流或单 Agent 循环。

## 5. 主要对照项目

Star 数量会持续变化，以下仅记录 2026-07-12 整理时的大致规模。对照项目的选择重点是代表性和学习价值，不是单纯排名。

### 5.1 MaxKB：第一对照项目

- GitHub：https://github.com/1Panel-dev/MaxKB
- 整理时约 2 万 Star。
- 定位：企业级智能体与知识库应用平台。
- 重点对比：
  - 知识库、文档、分段与会话的数据关系；
  - 文档上传、解析、向量化和索引流程；
  - RAG 检索与引用展示；
  - 模型供应商适配；
  - 后台任务与失败处理；
  - 工作流和 MCP 工具能力如何逐渐扩展。

MaxKB 与计划开发的项目功能距离较近，适合作为完成 MVP 后的主要源码对照对象。

### 5.2 FastGPT：产品流程与 RAG 对照

- GitHub：https://github.com/labring/FastGPT
- 定位：知识库问答、数据处理、RAG 和可视化工作流平台。
- 重点对比：
  - 创建知识库到开始问答的产品流程；
  - 文档处理和检索配置如何呈现给用户；
  - 数据集管理与问答效果调优；
  - 可视化工作流的节点和数据传递设计。

### 5.3 Dify：大型 AI 应用平台架构对照

- GitHub：https://github.com/langgenius/dify
- 整理时约 14 万 Star。
- 定位：包含模型管理、RAG、工作流、Agent、插件和可观测性的 AI 应用开发平台。
- 重点对比：
  - 大型项目如何划分领域和模块；
  - 应用、模型、数据集、工作流和插件之间的边界；
  - 模型配置、密钥管理与供应商扩展机制；
  - 异步任务、事件、缓存和部署结构；
  - 从原型走向生产系统后增加了哪些复杂性。

Dify 规模很大，不适合作为入门教程从头通读。应在自己的项目完成后，带着具体问题定向阅读。

### 5.4 RAGFlow：文档处理与检索专项对照

- GitHub：https://github.com/infiniflow/ragflow
- 定位：侧重文档理解、数据摄取、检索增强生成和 Agent 能力的 RAG 引擎。
- 重点对比：
  - 不同文档的解析方式；
  - 分段、索引和检索策略；
  - 检索结果质量与可解释性；
  - 复杂 RAG 流程如何组织；
  - 为什么生产级文档处理需要较多基础设施。

RAGFlow 更适合在遇到“能检索但效果不好”的问题后专项研究。

### 5.5 Langfuse：可观测性与评估专项对照

- GitHub：https://github.com/langfuse/langfuse
- 整理时约 2.8 万 Star。
- 定位：大模型调用追踪、评估、指标、提示词和数据集管理平台。
- 重点对比：
  - 一次 AI 请求应该记录哪些信息；
  - Trace、会话、模型调用和用户反馈之间的关系；
  - 如何建立评估数据集；
  - 如何观察质量、耗时、Token 和成本。

Langfuse 不是同类完整业务应用，而是用于补充生产级 AI 工程能力的专项参考。

## 6. 开发与对比方法

### 阶段一：只看边界，不看实现

- 阅读对照项目的 README、功能介绍和公开演示。
- 了解行业常见功能，确定自己的 MVP 边界。
- 暂时不深入阅读核心源码，避免直接复制成熟项目的设计。

### 阶段二：独立设计和实现

- 自己定义业务流程、数据模型、接口和模块。
- 使用 Codex 协助调研、编码、测试和排错。
- 对 Codex 生成的关键代码进行解释和验证。
- 保留早期设计和遇到的问题，不因设计不成熟而频繁推倒重来。

### 阶段三：带着问题阅读开源项目

完成 MVP 后，根据实际遇到的问题定向比较：

- 对照项目如何兼容不同模型？
- 为什么文档处理需要后台任务或队列？
- 对话、消息、知识库、文档和应用之间是什么关系？
- 如何处理流式输出的中断、取消与异常？
- 如何重试失败请求而不产生重复数据或重复费用？
- 如何记录 Token、耗时、错误、费用和用户反馈？
- 如何隔离不同用户的数据与密钥？
- 哪些复杂设计是业务需要，哪些是平台规模带来的成本？

### 阶段四：形成复盘

每个重要模块使用以下格式记录：

```text
问题：
我的设计：
开源项目的设计：
两者差异：
差异产生的原因：
我的设计在当前规模下是否足够：
准备如何改进：
```

最终形成一份《个人 AI 应用与开源项目架构对比复盘》，作为作品集的一部分。

## 7. 完成标准

项目“完成”不以代码量或功能数量衡量。MVP 至少应满足：

- 新用户能够按照说明启动项目并完成一次知识库问答。
- 回答能展示来自上传资料的引用来源。
- 模型回答失败时，系统能够给出可理解的错误信息。
- API Key 不会出现在前端代码、日志明文或 Git 仓库中。
- 能查看一次请求使用的模型、耗时、Token 和状态。
- 有一组固定问题用于验证回答是否基本符合预期。
- 有清晰的 README、系统结构说明和本地启动方式。
- 开发者本人能够解释一次请求从前端到模型再返回的完整流程。

## 8. API 使用边界与安全要求

- API Key 仅保存在后端环境变量或专用密钥配置中。
- `.env`、真实密钥和包含隐私的测试资料不得提交到 GitHub。
- 配置请求超时、有限重试、并发限制和费用预警。
- 日志默认不记录完整密钥和敏感文档内容。
- 明确区分大语言模型 API 与 Embedding API；技术选型时确认供应商是否同时提供。
- 如果资料会发送至第三方模型服务，需要在界面或文档中说明数据流向。
- 工具调用必须采用白名单和参数校验，不允许模型直接获得任意系统权限。

## 9. 新会话需要讨论的技术选型

下一次会话应基于本文继续确定以下内容：

1. 后端继续使用 Java/Spring Boot，还是使用 Python/FastAPI 等技术。
2. 前端采用何种方案，以及第一版是否需要前后端分离。
3. 使用哪一家大模型 API，是否兼容 OpenAI 接口。
4. 使用哪一种 Embedding 服务。
5. 数据库选择，以及向量数据使用独立向量数据库还是 PostgreSQL + pgvector。
6. 文档解析方案和第一版支持的文件类型。
7. 同步处理还是引入后台任务；第一版是否需要消息队列。
8. 流式输出使用 SSE 还是 WebSocket。
9. 用户认证、密钥配置和数据隔离做到什么程度。
10. 是否从第一版接入调用追踪工具，或先自行记录最小指标。
11. 项目采用单体结构还是拆分服务。
12. 测试、部署、费用预算和开发周期。

## 10. 技术选型原则

- 第一版优先完成闭环，不追求平台化。
- 主要学习变量应是 AI 应用开发，不同时引入过多陌生基础设施。
- 选择成熟、有文档、有社区、便于替换的组件。
- 不因为开源项目使用某项技术就盲目跟随，要结合个人项目规模判断。
- 把大模型视为不稳定、可能超时、可能产生错误内容且按量计费的外部依赖。
- 所有关键技术选择都记录“选择理由、替代方案和暂不选择的原因”。
