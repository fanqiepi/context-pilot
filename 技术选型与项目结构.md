# AI 应用项目技术选型与项目结构

> 更新日期：2026-07-13  
> 文档状态：MVP 技术选型已确认  
> GitHub 仓库：https://github.com/fanqiepi/context-pilot.git

| 项目名称 | GitHub 仓库名 | 项目定位 |
| --- | --- | --- |
| ContextPilot | `context-pilot` | 基于 Spring AI 的知识库与任务助手 |

## 1. 技术选型

### 1.1 后端

| 类别 | 选型 |
| --- | --- |
| 开发语言 | Java 21 LTS |
| 应用框架 | Spring Boot 4.1.x |
| AI 应用框架 | Spring AI 2.0.x |
| Web 框架 | Spring MVC；引入 WebFlux 响应式能力承接 Spring AI 流式输出 |
| 流式协议 | Reactor `Flux` + SSE |
| 业务数据访问 | MyBatis-Plus 3.5.x，使用 Spring Boot 4 Starter |
| 项目构建 | Maven |
| Java 基础包 | `io.github.fanqiepi.contextpilot` |
| 接口文档 | springdoc-openapi |
| 数据库迁移 | Flyway |
| 可观测性 | Spring Boot Actuator + Micrometer |
| 测试 | JUnit 5 + Mockito + Testcontainers |

业务表的 CRUD 与自定义 SQL 使用 MyBatis-Plus；向量表的写入和相似度检索使用 Spring AI `PgVectorStore`，不通过 MyBatis-Plus 操作向量。

### 1.2 模型服务

| 类别 | 选型 |
| --- | --- |
| Chat Model 供应商 | DeepSeek 官方 API |
| Spring AI 集成 | `spring-ai-starter-model-deepseek` |
| 默认 Chat Model | `deepseek-v4-flash` |
| 高质量对照模型 | `deepseek-v4-pro`，仅在评估或复杂问题中按需使用 |
| Chat API Key | 后端环境变量 `DEEPSEEK_API_KEY` |
| Embedding 供应商 | 阿里云百炼（DashScope） |
| Embedding Model | `text-embedding-v4` |
| Embedding 维度 | 1024，MVP 期间保持固定 |
| Embedding 集成 | Spring AI OpenAI Embedding 客户端，通过 OpenAI 兼容接口接入 |
| Embedding API Key | 后端环境变量 `DASHSCOPE_API_KEY` |

不使用即将停用的 `deepseek-chat` 和 `deepseek-reasoner` 旧模型别名。

Chat Model 和 Embedding Model 独立配置。文档分段会发送到百炼生成向量；检索命中的分段会发送到 DeepSeek 生成回答。README 和产品界面需要说明这一数据流向。

### 1.3 数据存储

| 类别 | 选型 |
| --- | --- |
| 关系数据库 | PostgreSQL |
| 数据库名称 | `context_pilot` |
| 向量存储与检索 | pgvector |
| Spring AI 集成 | PgVectorStore |
| 向量距离度量 | Cosine Distance |

PostgreSQL 同时保存业务数据和向量数据，MVP 不引入独立向量数据库。

### 1.4 文档处理

| 类别 | MVP 选型 |
| --- | --- |
| 文件类型 | TXT、Markdown、文本型 PDF |
| TXT 读取 | Spring AI `TextReader` |
| Markdown 读取 | Spring AI `MarkdownDocumentReader` |
| PDF 读取 | Spring AI `PagePdfDocumentReader`，保留页码元数据 |
| 文档切分 | Spring AI `TokenTextSplitter` |
| 原始文件存储 | 本地文件系统 `data/uploads/`，通过 `StorageService` 抽象访问 |
| 后台处理 | Spring `TaskExecutor` + `@Async` |
| 任务状态 | PostgreSQL 持久化处理状态、错误和重试次数 |

MVP 不支持扫描版 PDF、OCR、复杂表格结构恢复和 Office 文档。文档处理采用有限大小的本地线程池，不引入 RabbitMQ、Kafka 等消息队列。

### 1.5 前端

| 类别 | 选型 |
| --- | --- |
| 前端框架 | Vue 3 |
| 开发语言 | TypeScript |
| 构建工具 | Vite |
| UI 组件库 | Element Plus |
| 路由 | Vue Router |
| 状态管理 | Pinia |
| 普通 HTTP 请求 | Axios |
| SSE 客户端 | `@microsoft/fetch-event-source`，支持 POST 流式请求 |
| 包管理器 | npm |
| HTTP 通信 | REST API + SSE |

MVP 按单用户应用设计，不实现注册、登录、角色和权限管理；模型密钥不允许从前端配置或传输。

### 1.6 开发与部署

| 类别 | 选型 |
| --- | --- |
| 本地依赖编排 | Docker Compose |
| 应用结构 | 前后端分离的模块化单体 |
| 配置管理 | Spring Profile + 环境变量 |
| API Key 管理 | 仅保存在后端环境变量中 |
| MVP 文件存储 | 本地文件系统；运行数据不提交 Git |
| 后台任务 | 应用内线程池；不引入消息队列 |
| 用户体系 | 单用户，无认证；后续迭代再增加数据隔离 |
| 外部追踪平台 | MVP 不接入 Langfuse，先自行记录最小指标 |

### 1.7 AI 工程资产

| 类别 | 约定 |
| --- | --- |
| Codex 项目指令 | 根目录使用 `AGENTS.md`；仅在子目录规则确有差异时增加嵌套文件 |
| 提示词管理 | 提示词以文本文件形式保存在后端资源目录并纳入版本控制 |
| 评估资产 | 固定问题集、参考答案、测试文档和评估配置统一放入 `evals/` |
| 架构决策 | 重要技术决策使用 ADR 记录 |
| 持续集成 | GitHub Actions 执行后端测试、前端检查和密钥泄漏防护 |

## 2. 项目结构

### 2.1 总体结构

```text
context-pilot/
├── AGENTS.md                  # Codex 等编码代理的项目约束
├── README.md                  # 项目入口、能力说明和快速开始
├── LICENSE                    # 开源许可证
├── SECURITY.md                # 安全边界和漏洞报告方式
├── backend/                   # Spring Boot 后端
├── frontend/                  # Vue 前端
├── evals/                     # AI 评估数据、用例和配置
├── data/
│   └── uploads/               # 本地上传文件，整体加入 .gitignore
├── deploy/                    # Docker Compose 与部署配置
├── docs/                      # 需求、架构、接口和开发文档
├── .github/                   # CI、Issue 和 PR 模板
├── .env.example              # 环境变量示例，不包含真实密钥
├── .gitignore
└── docker-compose.yml         # 本地开发依赖编排入口
```

### 2.2 后端结构

后端采用模块化单体，优先按业务功能组织代码。

```text
backend/
├── pom.xml
└── src/
    ├── main/
    │   ├── java/io/github/fanqiepi/contextpilot/
    │   │   ├── common/            # 通用配置、异常和基础类型
    │   │   ├── knowledgebase/     # 知识库管理
    │   │   ├── document/          # 文档上传、解析、切分和索引
    │   │   ├── retrieval/         # 向量检索
    │   │   ├── chat/              # 会话、消息和 RAG 问答
    │   │   ├── model/             # Chat Model 与 Embedding 调用
    │   │   ├── observation/       # 模型调用记录和指标
    │   │   └── feedback/          # 回答反馈
    │   └── resources/
    │       ├── db/migration/      # Flyway 数据库迁移脚本
    │       ├── prompts/           # 提示词模板
    │       └── application.yml
    └── test/                      # 单元测试和集成测试
```

各业务模块根据需要包含接口层、应用层、领域模型和基础设施实现，不为简单功能强制创建空层级。

### 2.3 前端结构

```text
frontend/
├── package.json
├── vite.config.ts
└── src/
    ├── api/                       # REST 与 SSE 请求封装
    ├── components/                # 通用组件
    ├── composables/               # 流式问答等组合式逻辑
    ├── views/                     # 页面
    ├── stores/                    # 状态管理
    ├── router/                    # 前端路由
    ├── types/                     # TypeScript 类型
    └── utils/                     # 通用工具
```

### 2.4 AI 工程资产结构

```text
evals/
├── datasets/                      # 固定问题、参考答案和评估标签
├── fixtures/
│   └── documents/                 # 可公开提交的测试文档
├── configs/                       # 检索、生成和评分配置
└── reports/                       # 可选的基线报告，不提交敏感原始结果

backend/src/main/resources/prompts/
├── rag-system.md                  # RAG 系统提示词
└── citation-instructions.md       # 引用输出约束
```

提示词和评估数据必须可以追踪版本。真实用户资料、私密文档和包含敏感内容的运行日志不得作为评估样本提交。

### 2.5 文档结构

```text
docs/
├── requirements/
│   └── product-requirements.md    # 产品需求与 MVP 验收标准
├── architecture/
│   ├── technical-architecture.md  # 总体技术架构
│   └── decisions/                 # Architecture Decision Records
│       └── 0001-use-pgvector.md
├── data/
│   └── data-model.md              # 领域模型和数据库设计
├── api/
│   └── api-design.md              # REST API、SSE 事件和错误约定
├── evaluation/
│   └── evaluation-plan.md         # RAG 固定问题集和质量指标
└── development/
    ├── local-development.md       # 本地环境与启动方式
    └── deployment.md              # 部署、配置和回滚说明
```

文档正文可以使用中文，目录名和文件名使用英文。接口实现后，OpenAPI 作为接口字段的主要事实来源，`api-design.md` 重点记录跨接口约定和设计理由。

### 2.6 GitHub 工程结构

```text
.github/
├── workflows/
│   └── ci.yml                     # 后端测试、前端检查和构建
├── ISSUE_TEMPLATE/                # Bug 与功能请求模板
└── pull_request_template.md       # PR 检查项
```

个人 MVP 初期不强制创建 `CODE_OF_CONDUCT.md`、`GOVERNANCE.md` 和 `SUPPORT.md`。当项目开始接受外部贡献时，再补充 `CONTRIBUTING.md` 和社区治理文件。

### 2.7 Codex 指令结构

根目录 `AGENTS.md` 记录全项目通用规则，包括项目边界、技术栈、目录职责、验证要求和安全约束。后端或前端出现专属构建命令与规范后，可分别增加：

```text
backend/AGENTS.md
frontend/AGENTS.md
```

子目录文件只写差异，不重复根目录全部内容，避免为 Codex 注入过多上下文。

### 2.8 核心模块关系

```text
Vue 前端
    │ REST / SSE
    ▼
Spring Boot 模块化单体
    ├── 知识库与文档模块 ──► 百炼 text-embedding-v4
    ├── 检索模块 ──────────► PgVectorStore
    ├── 会话与问答模块 ────► DeepSeek API
    └── 调用记录与反馈模块
                    │
                    ▼
          PostgreSQL + pgvector
```

### 2.9 MVP 外部服务与数据流

```text
文档上传
    → 本地文件存储
    → 解析与分段
    → 百炼 Embedding API
    → PostgreSQL + pgvector

用户提问
    → 百炼 Embedding API
    → pgvector 相似度检索
    → DeepSeek API
    → SSE 流式回答与引用
    → PostgreSQL 保存会话、调用记录和反馈
```

MVP 依赖两个外部模型服务，但应用层分别面向 Spring AI `ChatModel` 和 `EmbeddingModel` 接口，不允许在业务模块中散落供应商专用 HTTP 调用。
